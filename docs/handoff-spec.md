# MMM Handoff Spec

## Purpose

This document is the primary handoff brief for any future agent or developer taking over MMM (Monday Music Mix).

MMM is a data-first, static, local-first editorial music archive. It is intentionally not a SaaS app, not a CMS-backed web product, and not a cloud automation project. The core philosophy is:

- content lives in versioned JSON files
- editorial workflows run on Aditya's local machine
- GitHub Pages only publishes the built static output
- the UI should feel like an honest music archive, not a fake brand universe

A new agent should read this file first, then:
- `README.md`
- `docs/operator-flow.md`
- `docs/data-import-assumptions.md`
- `docs/reference-design/stitch/README.md`

## Product intent

MMM exists to reboot the old Monday Music Mix Tumblr archive as a durable site that can:

1. preserve imported historical mixes
2. generate and draft new mixes locally
3. let an editor review, annotate, approve, and publish them
4. present mixes, notes, and archive context in a tasteful static site
5. remain maintainable by future agents without hidden system state

## Non-goals

These are important because future agents may try to overbuild:

- no database
- no auth system
- no server-side app runtime
- no hosted editorial workflow in GitHub Actions
- no dependency on OpenAI or other hosted AI for the core pipeline
- no fake playback claims when real audio sources do not exist
- no “music startup” product bloat like accounts, likes, comments, feeds, or dashboards for vanity

## Current architecture summary

MMM has three layers:

1. Content/data layer
- JSON files in `data/`
- schemas in `schemas/`
- imported Tumblr legacy source data preserved for provenance

2. Editorial tooling layer
- Python scripts in `scripts/`
- validation
- template generation
- draft generation
- publishing
- taste-profile building
- import pipeline

3. Presentation layer
- static site generated by `scripts/build.js`
- styles in `src/static/site.css`
- output in `dist/`
- deploy through GitHub Pages

## Runtime and deployment model

Authoritative workflow split:

Local machine responsibilities:
- create content
- generate weekly drafts
- validate content
- review/edit content
- approve/publish content
- optionally preview locally

GitHub responsibilities:
- run tests on push
- build static site
- deploy `dist/` to GitHub Pages

This split is intentional and should be preserved unless Aditya explicitly changes it.

## Functional requirements

### FR1 — Historical import
The system must be able to import old Monday Music Mix Tumblr RSS content into structured JSON.

Current implementation:
- yes

Required behavior:
- preserve source URL / provenance
- preserve legacy HTML snapshot where available
- parse tracklists
- preserve favorite-track signals from bolded Tumblr items
- preserve timestamps and mix numbering when inferable

### FR2 — Static archive browsing
The site must present published mixes in a browsable archive.

Current implementation:
- yes

Required behavior:
- route: `/archive/`
- show title/date/excerpt
- show archive relationships like note counts / highlighted tracks where available
- allow lightweight client-side discovery/filtering

### FR3 — Mix detail pages
Each published mix must have a dedicated route.

Current implementation:
- yes

Required behavior:
- route: `/mixes/[slug]/`
- show metadata, tracklist, notes/context, related notes, adjacent mixes
- preserve provenance/source context
- avoid misleading playback UI

### FR4 — Notes index and note detail pages
Editorial notes must be first-class archive objects.

Current implementation:
- yes

Required behavior:
- route: `/notes/`
- route: `/notes/[slug]/`
- support note-to-mix relationships
- support lightweight discovery/filtering on the notes index

### FR5 — Local editorial dashboard
The site must expose a generated local-facing summary page to help orient an operator.

Current implementation:
- yes

Required behavior:
- route: `/studio/`
- summarize drafts, published mixes, notes, featured mix, recommended next actions, recent routes
- stay static/data-first
- feel tasteful, not admin-panel garbage

### FR6 — Draft generation
The system must support generating a new weekly draft locally.

Current implementation:
- yes, deterministic/local

Required behavior:
- no hosted AI dependency required
- local-safe default behavior
- write output to `data/drafts/`

### FR7 — Content validation
The system must validate content health before publish/deploy.

Current implementation:
- yes

Required behavior:
- validate site metadata
- drafts
- published mixes
- notes
- archive indexes
- notes indexes
- generated aggregate files
- produce actionable output

### FR8 — Publish flow
The system must publish approved drafts into published archive content.

Current implementation:
- yes

Required behavior:
- only approved mixes publish
- write into `data/published/`
- rebuild aggregate indexes
- optionally set featured mix

### FR9 — Local scheduling
Weekly editorial workflows should be runnable locally via LaunchAgent or direct shell invocation.

Current implementation:
- yes

Required behavior:
- scheduled runs must not overwrite active drafts blindly
- scheduled runs should not do unnecessary site builds
- scheduled runs should skip pytest by default, with an explicit override available
- logs should be durable enough for local inspection

### FR10 — Deploy to GitHub Pages
Push to `main` should deploy the static site.

Current implementation:
- yes

Required behavior:
- run tests
- build site
- deploy Pages
- no hosted editorial generation

## UX and design requirements

### UX1 — Tone
The product should feel:
- nocturnal
- editorial
- restrained
- human
- music-first

It should not feel:
- gamified
- “creator platform” generic
- fake luxurious
- needlessly theatrical
- app-store-template-ish

### UX2 — Honesty
The UI should never pretend it has playback certainty when it does not.

Examples:
- dead Mega links should not be treated as primary playback
- imported Tumblr art should not always be treated as canonical cover art
- fallback track search links are acceptable if clearly framed as fallback helpers

### UX3 — Discovery, not clutter
Archive and notes search/filter should remain lightweight and mostly invisible until needed.

### UX4 — Context density
Mix pages should prioritize:
1. title / metadata
2. tracklist
3. related notes / context
4. source and provenance

Not the reverse.

## Data requirements

### Mix draft shape
Editorial draft mixes in `data/drafts/` should support:
- slug
- title
- date
- status
- summary
- notes
- tags
- featured (optional)
- tracks[] with:
  - artist
  - title
  - why_it_fits

### Published mix shape
Published mixes in `data/published/` should preserve:
- source metadata
- mix numbering
- summary/intro
- tags
- tracklist
- favorite track signals
- optional listening/provider data
- optional cover credit
- legacy provenance snapshot

### Notes shape
Notes in `data/notes/` should support:
- id
- slug
- status
- title
- publishedAt
- summary
- body[]
- relatedMixSlugs[]
- tags[]

### Site metadata shape
`data/site.json` should define:
- site identity
- base URL
- repo URL
- author metadata
- nav structure
- featured mix slug

## Current strengths

The project is already strong in these areas:
- simple architecture
- strong local/editorial separation from deploy
- solid data-first workflow
- real tests
- real content validation
- static build determinism
- usable archive, notes, and studio routes
- reference design material preserved inside repo

## Current weaknesses / known rough edges

These are not fatal, but future agents should be aware:

1. provider/listening data is still sparse and partially reconstructed
2. imported historical content may contain minor RSS/Tumblr weirdness
3. studio/dashboard is useful but still basic
4. client-side discovery is intentionally light; not a full search engine
5. GitHub Actions still shows upstream Node 20 deprecation warnings on some actions despite successful deploys

## Operational commands

Use these as the canonical command set.

Validate content:
- `python3 scripts/validate_content.py`
- `npm run content:validate`

Create draft mix template:
- `python3 scripts/create_content.py draft-mix --date 2026-04-13`
- `npm run draft:new -- --date 2026-04-13`

Create note template:
- `python3 scripts/create_content.py note --title "A note title" --related-mix mix-036-thirtysixth`
- `npm run note:new -- --title "A note title" --related-mix mix-036-thirtysixth`

Generate weekly draft:
- `python3 scripts/generate_weekly_draft.py --mode auto`
- `npm run draft:generate`

Publish approved mix:
- `python3 scripts/publish_mix.py <slug-or-path> --feature`

Build site:
- `npm run build`

Preview locally:
- `npm run dev`

Run tests:
- `python3 -m pytest -q`

Run local scheduled workflow manually:
- `./scripts/run_local_workflow.sh --scheduled`
- `./scripts/run_local_workflow.sh --scheduled --run-tests`

Install the local LaunchAgent:
- `python3 ops/install_launch_agent.py`

## Rules for future agents

1. Do not introduce a database unless Aditya explicitly asks.
2. Do not move editorial automation into GitHub Actions.
3. Do not add hidden dependencies on hosted AI for core workflows.
4. Do not convert the site into a framework app just because you can.
5. Preserve the “honest archive” principle in playback, art, and metadata.
6. Prefer extending existing JSON/data flows over inventing parallel systems.
7. When touching data shapes, update:
   - schemas
   - tests
   - docs
   - any generators/validators affected
8. Keep the UI tasteful and minimal.

## Recommended next roadmap

Short-term next steps:
1. improve provider-quality heuristics for listening data
2. add more real note coverage for published mixes
3. add automatic index regeneration helpers for notes/archive maintenance
4. improve studio route with clearer validation output and route health summaries

Medium-term:
1. add optional richer provider embeds where trustworthy
2. add more structured taxonomy/tagging for archive discovery
3. improve draft-generation taste heuristics using imported archive signals
4. add import/repair tools for legacy data cleanup

Longer-term, only if clearly useful:
1. better reconstruction workflows for real playlist mirrors
2. richer editorial note linking/series organization
3. optional locally generated art workflows if Aditya wants them
